# YouTube A11y Agent — Chrome extension (MV3)

The installable form of the project. Instead of pasting snippets, it auto-injects the
**provider** (WebMCP tools) and the **agent** (voice + vision; deterministic commands by
default, optional AI replies via on-device Gemini Nano or your own Gemini API key) on
every YouTube page, and gives you a small popup to start/stop a hands-free conversation.

It reuses the exact same code as the userscripts — `provider.js` and `agent.js` are synced
from `src/` by `npm run build:extension` (don't edit them directly).

## What's inside

| File | World | Role |
|------|-------|------|
| `manifest.json` | — | MV3 manifest; declares the content scripts + popup + permissions |
| `agent.js` | MAIN | the consumer agent (synced from `src/agent/dev-agent.user.js`) |
| `provider.js` | MAIN | the WebMCP tool provider (synced from `src/youtube-a11y-agent.user.js`) |
| `agent-control.js` | MAIN | maps extension commands → `window.ytAgent`; **talk-first** announce on first interaction; installs the cloud transport |
| `bridge.js` | ISOLATED | relays popup commands AND cloud requests between the page and the extension (MAIN can't use `chrome.*`) |
| `service-worker.js` | — | dispatches the global `Alt+Shift+A` shortcut; holds the **BYOK key** (`chrome.storage.local`) and makes the Gemini API fetch |
| `popup.html` / `popup.js` | — | toolbar control panel + **API-key entry** — a **visual fallback**, not the primary entry |
| `icons/` | — | extension icons (generated by `npm run build:icons`) |

Why two content-script "worlds": `document.modelContext` (still `navigator.modelContext`
on Chrome 149) and the Prompt API only exist in the page's **MAIN** world, but `chrome.*`
messaging only works in the **ISOLATED** world — so the agent runs in MAIN and `bridge.js`
shuttles popup commands across.

## Optional AI replies — bring your own key (BYOK)

Direct commands (play, pause, search, browse, open N, captions…) are **deterministic and
never use any AI**. For everything else ("what's this video about?"), AI replies are **off
by default** and opt-in:

1. Create an API key at [Google AI Studio](https://aistudio.google.com/) — ideally under a
   dedicated **no-billing** Cloud project (the free quota then acts as a hard cost
   ceiling), with the key **API-restricted to the Generative Language API**. (HTTP-referrer
   restrictions don't work for extension service workers — leave application restriction
   "None".)
2. Open the extension popup → paste the key → **Save** → check **Enable AI replies**.
3. The key is stored in `chrome.storage.local` and read **only by the service worker**,
   which makes the API call — the key never enters the page, is never sent to the
   developer, and is removable with one click. Page CSP doesn't apply (the fetch happens
   in the worker). Default model: `gemini-3.1-flash-lite` (pinned stable; change with
   `ytAgent.setCloudModel()`).
4. Without a key, "Enable AI replies" has nothing to call, so conversational requests get a
   short "add a key" coaching reply — **direct commands still work with no AI**. The
   on-device Gemini Nano engine stays a console-only opt-in (`ytAgent.setEngine("nano")`),
   intentionally kept off the popup after it repeatedly froze the dev machine — see the
   kill-switch note in the project `CLAUDE.md`.

Privacy details: [`docs/store/privacy-policy.md`](../docs/store/privacy-policy.md).

## Requirements

Same Chrome flags as the rest of the project (enable, then relaunch):

- `chrome://flags/#enable-webmcp-testing`
- `chrome://flags/#prompt-api-for-gemini-nano`
- `chrome://flags/#optimization-guide-on-device-model` → "Enabled BypassPerfRequirement"
- `chrome://flags/#prompt-api-for-gemini-nano-multimodal-input` (for the vision feature)

## Install (load unpacked)

1. Run `npm run build:extension` once (generates `provider.js` + `agent.js`). They're also
   committed, so you can skip this unless you changed `src/`.
2. Open `chrome://extensions`, toggle **Developer mode** (top-right) on.
3. Click **Load unpacked** and select this `extension/` folder.
4. Pin the extension (puzzle-piece icon → pin) so the popup is one click away.

## Use — it talks first (no clicking required)

This is an accessibility supplement, so the primary flow is **hands-free, no sighted
interaction**:

1. Open <https://www.youtube.com>. The agent auto-loads on every page (re-injects on
   navigation — no re-pasting).
2. **On your first interaction with the page** (a keypress — e.g. Tab/arrows from a screen
   reader — or a click), the agent **speaks**: *"YouTube accessibility agent ready. Press
   the backtick key and speak to ask me anything; I'll respond when you pause."*
   (Browsers forbid audio on bare page-load, so the first interaction is the earliest it can
   legally speak.) It greets once per tab session.
3. **Tap the backtick key `` ` `` once and speak** to ask something — it sends automatically
   when you pause, and **press it again anytime to interrupt** the reply and start a new
   turn (the mic is never held open). You'll hear short tones: a rising beep when it starts
   listening, a higher beep when your voice is captured, and a soft beep when it's your turn
   again — so you're never waiting in silence. The agent also says what it's doing
   ("Searching.", "Opening."). Press **`Alt+Shift+A`** anytime for the full spoken overview
   of the page.
4. **Arrow keys browse the feed** (on the home page and search results): **Down/Up** move to
   the next/previous video and describe it ("Item 3 of 20: <title>, by <channel>, 10:26"),
   **Enter** plays it, **Escape** stops browsing. Arrows are ignored while you're typing in
   the search box, and on `/watch` (where they seek the player). The agent welcomes you back
   warmly when you're signed in (it uses your name only on the rare occasion YouTube exposes
   it), and on the home page reads out the available categories.

The **popup is a visual fallback** for sighted users / debugging — ▶ Start (continuous
loop), ■ Stop, 🔊 Greeting, and a Nano-transcription toggle. You can also drive everything
from the page console via `window.ytAgent` (e.g. `ytAgent.describeThumbnail(1)` for vision).

> Why a keypress and not autoplay on load: the browser's autoplay/gesture policy blocks
> speech and mic until the user interacts. Talking on first interaction is the accessible,
> policy-compliant version of "talk first."

## Notes & limitations

- **Conversation context now survives full navigations.** The last few spoken turns are
  persisted in the tab's `sessionStorage` and re-fed to the model on the next page, and
  `pend()` continuations announce search/open results — so flows feel continuous. (The
  model *session* itself is rebuilt per page; only the text history carries.)
- **Cloud requests go page → bridge → service worker.** `window.postMessage` is
  page-visible by design, so the relayed payload is the request text only — the key never
  crosses the bridge in either direction. A hostile script on youtube.com could submit a
  request through the relay (spending the user's quota), but could never read the key.
- **Editing logic:** change `src/…`, then `npm run build:extension`, then reload the
  unpacked extension in `chrome://extensions`. Icons: `npm run build:icons`.
- **Web Store packaging:** icons ✅, privacy policy ✅
  ([`docs/store/privacy-policy.md`](../docs/store/privacy-policy.md)), permission
  justifications ✅ ([`docs/store/permission-justifications.md`](../docs/store/permission-justifications.md)).
  Remaining: host the privacy policy at a public URL, screenshots/promo images, and a
  packed `.zip` upload.
